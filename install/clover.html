Clover

Introdução

O Clover é um bootloader originalmente desenvolvido para inicializar Hackintoshes, é um projeto já muito maduro e que disponibiliza binários para diversos tipos de soluções, poŕem um dos grandes diferenciais do Clover é a capacidade de emular o UEFI em sistemas de legado... Isso mesmo que você leu! Emular as capacidades do UEFI em sistemas de legado! Obviamente eu não recomendo isso para novatos por ser uma implementação de baixo nível e não ser tão estável quanto usar em BIOS um sistema instalado em um computador que tem BIOS... Mas é uma alternativa bastante interessante.

É bom avisar de antemão que o sucesso nessa implementação depende largamente da versão do Clover, dos loaders utilizados, e da compatibilidade dessa versão com a sua placa mãe. E certamente não é compatível com todos os casos, e ela quase nunca vai ser compatível com virtualização (o que também seria inútil de ser utilizado em virtualizações mas tudo bem).

Instalação (UEFI simulada em BIOS)

Eu te recomendo fortemente a fazer esse procedimento através de uma conexão SSH, você pode até mesmo fazer esse procedimento preparando o disco via terminal para um outro computador, então não se limite a fazer isso no computador alvo.

Antes de mais nada precisamos da ISO do Clover, você pode começar baixando a ISO do Clover através desse link aqui e preparando em um outro pendrive, ou mesmo colocando ela em um drive de CD (real ou virtual), e caso esteja dentro de um sistema live com suporte a git e curl pode usar o comando abaixo para baixar para seu pwd:

curl -L -O $(curl -s https://api.github.com/repos/CloverHackyColor/CloverBootloader/releases/latest | grep -o 'https://github.com/CloverHackyColor/CloverBootloader/releases/download/[^"]*/Clover-.*-X64.iso.7z')

Se você baixou o pacote ao invés de usar a ISO, extraia a ISO através do pacote 7zip com o comando:

7z x Clover-*-X64.iso.7z

Com a ISO e diretório prontos monte a ISO com o comando:

mount --mkdir -o loop Clover-*-X64.iso /mnt/iso

E caso tenha essa ISO em outro disco, seja numa máquina real ou virtual pode montar com o comando:

mount --mkdir -t iso9660 /dev/sr0 /mnt/iso

É bom lembrar que o Clover presume um layout de disco UEFI/GPT, a única diferença é que a a flag "Legacy BIOS bootable" deve ser marcada para a ESP. Então, vamos criar uma ESP válida para o Clover usando o GDISK:

Preparação para armazenamento em sistemas baseados em UEFI

Caso queira limpar a tabela de partição desse dispositivo (Leia-se apagar tudo!) pode usar o comando abaixo:

sgdisk -Z /dev/storageA

Para particionar o disco utilizaremos a ferramenta gdisk na unidade desejada, para isso utilize o comando:

gdisk /dev/storageA

Com o gdisk aberto na unidade desejada criaremos uma nova tabela de partição em GPT com o comando:

o

Partição ESP (EFI system partition)

Para uma instalação de sistema em modo UEFI precisamos criar uma partição ESP, que é uma partição específica para ser acessada pela placa mãe para iniciar os sistemas. Para criar a partição ESP use o comando:

n

O sistema irá exibir a seguinte linha para que você escolha o número da partição:

partition number: default

Apenas tecle Enter.

Logo em seguida o sistema irá exibir a seguinte linha para que você escolha o ponto de início da partição:

first sector: default

Apenas tecle Enter.

Logo em seguida o sistema irá exibir a linha abaixo para que você escolha o ponto de término da partição e delimite o seu tamanho, por questões de compatibilidade recomendo 550M, sendo assim deveremos utilizar o comando +550M e teclar Enter, de maneira que o diálogo no terminal fique da seguinte forma:

last sector: +550M

Logo em seguida o sistema irá exibir a linha abaixo para que você escolha o tipo da partição, para o tipo ESP devemos digitar o tipo ef00 e teclar Enter de maneira que o diálogo no terminal fique da seguinte forma:

Hex code or GUID (L to show codes, Enter = 8300): ef00

Com isso temos uma partição no início do armazenamento, de 550M e do tipo EFI system partition.

Adição de atributos na ESP

Assim que retornar a tela de inserção de comandos (Command (? for help):), vamos acessar as opções avançacas de recovery com o comando r:

r recovery and transformation options (experts only)

Acessar o menu de funcionalidade extra com o comando x:

x extra funcionality (experts only)

Agora vamos setar o atributo Legacy BIOS bootable com o comando a (caso pergunte, informe o número da partição ESP):

a set attributes

E vamos inserir a opção 2:

2: legacy BIOS bootable

Como não há mais nenhum atributo para ser inserido, apenas tecle Enter e depois insira o comando w para gravar as alterações.

Caso esteja usando o Parted você pode usar o comando abaixo para deixar a partição com o atributo legacy BIOS bootable ativo:

sudo parted /dev/storageAP1 set 1 legacy_boot on

Caso queira verificar a flag com o GDISK, acesse as opções avançacas de recovery com o comando r:

r recovery and transformation options (experts only)

Em seguida acesse as informações detalhadas da partição com o comando i, e informe o número da partição caso perguntado:

i show detailed information on a partition

Se o o campo de atributos estiver como no exemplo abaixo está correto:

Attribute flags: 0000000000000004

Caso esteja dessa outra forma, insira a flag:

Attribute flags: 0000000000000000

Caso queira usar o Parted para isso pode usar o comando:

sudo parted /dev/storageA print

E na seção flag você poderá ver a flag legacy_boot.

Formatando a partição ESP

Por padrão é formatada em FAT32... Mas para esse caso específico, considerando que não estamos lidando com uma placa mãe com todas as padronizações garantidas para um UEFI nativo, a partição ESP varia conforma o suporte da sua placa mãe, as chances são que ela AINDA deve ser em FAT32, mas pode ser por exemplo exFAT ou talvez até HFS para um Mac real. Se abrir a pasta BootSectors do seu Clover vai achar arquivos de suporte para HFS, FAT32 e exFAT (e até variações desses arquivos para redirecionamento de boot), fique a vontade pare tentar com eles (e recomendo que você veja a descrição de cada um na documentação). Mas supondo que por acaso a sua partição ESP é a /dev/storageAP1, utilize o comando abaixo para formatar a partição substituindo o /dev/storageAP1 para o devido endereço da partição ESP no seu dispositivo:

mkfs.fat -F32 /dev/storageAP1
Formatação avançada (Clique para ver)

Diferentes sistemas de arquivo (e até mesmo ferramentas) podem ter por padrão o uso de tamanhos distintos de setor. Caso você queira especificar o tamanho do setor utilize o argumento -S como no exemplo abaixo:

mkfs.fat -F 32 -S 4096 /dev/storageAP1

Embora não seja estritamente necessário, é recomendado o uso de rótulos para identificar as partições para o caso de você não ter em mente nenhum outro identificador. Caso você queira especificar o rótulo da partição adicione o argumento -n seguido do "ROTULO" como no exemplo abaixo:

mkfs.fat -F 32 -n "ESP" /dev/storageAP1

Descrição das Master Boot Records

Segue abaixo as descrições dos loaders de estágio 1:

  • boot0af (boot0 Active First) - Setor MBR que busca a partição ativa na tabela MBR (e depois na tabela GPT). Originado do projeto boot132 da Apple. Esta versão do boot0af implementa suporte ao esquema híbrido de partições GUID/MBR. Escrito por Tamás Kosárszky em 10/03/2008;

  • boot0ss (boot0 Signature Scanning) - Setor MBR que busca por qualquer partição com uma assinatura PBR válida, independentemente de estar ativa ou não. Útil para o Windows, que exige que sua partição esteja ativa. Compartilha o mesmo código do boot0af, mas a ordem de busca é diferente. Escrito por JrCs em 08/05/2013;

  • boot0md (boot0 Multiple Drives) - Setor MBR que busca por todos os discos e busca por uma partição inicializável (com boot1h) e a carrega, se não for encontrada, itera sobre os discos novamente e busca por uma partição ativa e a carrega;

Descrição das Partition Boot Records

Segue abaixo as descrições dos loaders de estágio 2:

  • boot1h - Setor PBR para partição formatada em HFS+. Procura pelo arquivo "boot" na raiz da partição. Originado do projeto boot132 da Apple. Escrito por Tamás Kosárszky em 14/04/2008. Modificado por Slice para suportar arquivos de boot maiores. Não mais 440 KB como a versão original, mas 472 KB necessários para inicializar o Clover-64;

  • boot1h2 - Setor PBR para partição HFS+, com boot alternativo baseado na tecla pressionada. Arquivo para inicializar = "boot{key}". Codificado por dmazar com base no boot1h;

  • boot1f32 - Setor PBR para partição formatada em FAT32. Procura pelo arquivo "boot" na raiz da partição. Útil para partições EFI ou pendrive. Escrito por mackerintel em 26/01/2009;

  • boot1f32alt - Setor PBR para partição FAT32, com boot alternativo baseado na tecla pressionada. Arquivo para inicializar = "boot{key}". Modificado por Slice com base no boot1f32 e boot1h2;

  • boot1x - Setor PBR para partição formatada em exFAT. Procura pelo arquivo "boot" na raiz da partição. Útil para partições EFI ou pendrive. Escrito por Zenith432 em 19/11/2014;

Descrição das Partition Boot Records

Segue abaixo as descrições dos loaders de estágio 3:

  • boot6 - CloverEFI 64 bits - Com driver ATA/ATAPI próprio;

  • boot7 - CloverEFI 64 bits (padrão) - Inclui um driver BiosBlockIO que funciona com qualquer controlador compatível com o BIOS. Útil para implementações bugadas de BIOS ou para cenários incomuns;

É bom lembrar que atualmente o Clover só tem lançamentos em 64 bits, dependendo do quão antiga é a sua placa mãe pode ser que você precise encontrar um estário 3 de 32 bits.

Preparando o disco para o Clover

Vamos começar com uma cópia do primeiro setor da partição ESP (Partition Boot Record) e armazenar no diretório /tmp/original_PBR:

dd if=/dev/storageAP1 of=/tmp/original_PBR bs=512 count=1 conv=notrunc

Vai resultar em usa saída como:

1+0 records in
1+0 records out
512 bytes copied

Agora vamos copiar o arquivo /mnt/iso/usr/standalone/i386/boot1f32 para /tmp/new_PBR:

cp /mnt/iso/usr/standalone/i386/boot1f32 /tmp/new_PBR

Agora vamos copiar exatos 87 bytes da Partition Boot Record original para a nova Partition Boot Record:

dd if=/tmp/original_PBR of=/tmp/new_PBR skip=3 seek=3 bs=1 count=87 conv=notrunc

Vai resultar em usa saída como:

87+0 records in
87+0 records out
87 bytes copied

Agora vamos regravar a nossa Partition Boot Record do disco com esse novo arquivo recém gerado:

dd if=/tmp/new_PBR of=/dev/storageAP1 bs=512 count=1 conv=notrunc

Vai resultar em usa saída como:

1+0 records in
1+0 records out
512 bytes copied

E agora vamos gravar a nossa Master Boot Record com o BOOT0 do arquivo boot0ss:

dd if=/mnt/iso/usr/standalone/i386/boot0ss of=/dev/storageA bs=440 count=1 conv=notrunc

Vai resultar em usa saída como:

1+0 records in
1+0 records out
440 bytes copied

Instalando o Clover no disco

Para desencargo de consciência vamos montar a partição ESP como /mnt/efi:

mount --mkdir /dev/storageAP1 /mnt/efi

Agora vamos copiar os arquivos do Clover para a partição ESP:

cp -r /mnt/iso/efi /mnt/efi/ cp /mnt/iso/usr/standalone/i386/x64/boot6 /mnt/efi/boot

Por fim demonte as partições com o comando:

umount /mnt/efi /mnt/iso

Um detalhe importante...

Obviamente que o após esse processo de instalação, o /efi/EFI/BOOT/BOOTX64.EFI vai ser o loader do Clover... Mas nada impede você de substituir pelo rEFInd ou qualquer outro desde que ele não presuma um comportamento complexo e/ou de baixo nível do firmware UEFI. Sendo assim fique a vontade para usar esse load do Clover para usar o seu bootloader preferido.
